## 1. 프로젝트 배경

UMC Product Web은 운영팀이 정책, 계정, 데이터 관리를 한 화면에서 처리하기 위한 백오피스입니다.  
이 프로젝트에서 내가 집중한 핵심은 다음 두 가지였습니다.

- 운영 도메인이 늘어나도 흔들리지 않는 **확장 가능한 구조**
- 내부 서비스라도 배포 품질을 유지하는 **검증 가능한 개발 프로세스**

2026년 2월 23일 기준으로 저장소 기여 589 commits를 기록했고, 단일 프론트엔드 메인 기여자로 라우팅/데이터/폼/품질 체계를 정리했습니다.

## 2. 라우팅과 서버 상태를 먼저 표준화

운영툴은 화면 수가 빠르게 늘어납니다.  
초기부터 TanStack Router와 TanStack Query를 표준으로 두고, 라우트 책임과 데이터 책임을 분리했습니다.

- 라우트: 화면 단위 책임(권한, 진입 경로, 레이아웃)
- 쿼리: 서버 상태 책임(캐시, 갱신, 오류 처리)

이 분리를 먼저 해두면 신규 메뉴를 추가할 때도 규칙을 그대로 재사용할 수 있어, 기능이 늘어날수록 오히려 개발 속도가 유지됩니다.

## 3. 폼은 RHF + Zod로 강제된 일관성 확보

백오피스 폼은 입력 조건이 자주 바뀌고 예외 케이스가 많습니다.  
React Hook Form + Zod 조합을 기본값으로 고정해, 다음을 얻었습니다.

- UI 입력과 검증 규칙의 일치
- 에러 메시지의 일관된 렌더링
- 신규 폼 개발 시 복붙 가능한 공통 패턴

결과적으로 폼 페이지가 늘어나도 “매번 다른 스타일의 검증 코드”가 생기지 않게 막을 수 있었습니다.

## 4. 품질은 사람 의존이 아니라 파이프라인으로

협업 속도를 올리려면 코드 리뷰 이전에 기본 품질을 자동으로 거르는 장치가 필요했습니다.

- Storybook + Chromatic: 컴포넌트 단위 시각 회귀 체크
- Vitest: 유닛/로직 테스트
- Playwright: 핵심 플로우 E2E 검증
- ESLint/Prettier/Husky/Commitlint: 포맷과 컨벤션 자동화

이 조합 덕분에 리뷰 단계에서는 스타일 지적보다 비즈니스 로직 검토에 집중할 수 있었습니다.

## 5. 회고

운영 백오피스는 “화려한 UI”보다 “오래 버티는 구조”가 더 중요합니다.  
이번 프로젝트에서 가장 큰 수확은 기능 구현 자체보다, 확장과 유지보수를 전제로 한 프론트엔드 구조를 실전으로 운영해본 경험이었습니다.
